Initial p50 values:

- `/readyz`: bounded `1957.490644 ms`, enhanced `2025.221738 ms` (`+3.46%`).
- Max RSS: bounded `483.9296875 MiB`, enhanced `472.58984375 MiB` (`-2.34%`).
- `plugins.runtime-post-bind.total`: bounded `489.3 ms`, enhanced `1026 ms` (`+109.7%`).
- `plugins.bootstrap.total`: bounded `256.6 ms`, enhanced `247.8 ms` (`-3.43%`).
- `sidecars.plugin-services.total`: bounded `1051.9 ms`, enhanced `1142.9 ms` (`+8.65%`).

Trace finding: `plugins.runtime-post-bind.total` is **not** the bind duration. It is
the timeline position, measured from gateway startup, at which the
`plugins.runtime-post-bind` span ended. The actual span duration is the unsuffixed
`plugins.runtime-post-bind` value. This explains why the old `+536.7 ms` `.total`
delta did not translate into `/readyz` latency: `.total` includes all earlier
startup phases and can move for reasons outside plugin bind work.

Initial per-plugin trace attributed the enhanced delta to
`plugins.gateway-load.plugin.native-memory-citations.registerMs` (`~548.6-564.2 ms`
enhanced vs `~0.4-0.5 ms` bounded). A CPU profile of an enhanced package-style boot
showed the expensive work was module/schema dependency loading and resolution,
especially duplicate `typebox` evaluation for static schemas, not graph extraction,
snapshot building, observation append, model loading, memory indexing, or any
tool-result computation.

Disposition: **remove avoidable schema dependency load; keep bind behavior otherwise**.

Change: static config/tool schemas now use plain JSON Schema objects and the package
no longer has a runtime `typebox` dependency. This preserves tool outputs and mode
semantics while removing an avoidable startup dependency from the plugin entry.

Classification:

- Plugin module/static schema load (`~371.6 ms` enhanced p50 after change):
  registration-path dependency load; not eager graph/model/index work.
- Three bounded core tool registrations (`~0.4 ms`): registration, inherent.
- Two enhanced optional tool registrations plus typed lifecycle hooks (`~29.5 ms`
  enhanced p50): registration, inherent to enhanced surface.
- Graph extraction, snapshot read/write, observation append/capping, and model-based
  extraction: not run during bind.

Hot-path guardrail: first enhanced `native_memory_search` after `/readyz` returned
one cited hit in `598.6 ms` including WebSocket setup, with no new lazy initialization
or turn-blocking work moved onto first use.

Gate policy:

- `/readyz` p50 regression remains the primary startup gate
  (`readyzP50RegressionPctBudget: 5`).
- `plugins.runtime-post-bind` span is a secondary watch with enhanced p50 budget
  `700 ms`.
- `plugins.runtime-post-bind.total` is demoted to diagnostics because it is a
  timeline position, not bind duration.

## Graceful-Degradation Waiver

Runtime negative testing on an external CLI runner was blocked by host model policy:
`claude-cli/claude-sonnet-4-6` is not allowed for agent `main`. This is recorded as a
deferred live-runtime test, not a silent skip.

Provider-neutral property verified for this release:

- Bounded mode registers no hooks, so it is identical across embedded runners,
  external CLI runners, and non-dispatching harnesses.
- Enhanced hook-dependent paths are opt-in and experimental. If a harness does not
  dispatch in-process lifecycle hooks, the hooks simply do not run.
- Unit coverage simulates enhanced hooks being registered but never fired and proves
  the three core tools (`native_memory_search`, `native_memory_fetch`,
  `native_memory_answer`) still succeed without throwing.
- Documentation now describes hook dispatch generically rather than naming a specific
  provider as the required negative case.

Re-run the live external-runner negative when a model/runner allowlist permits a
controlled `claude-cli`, Codex CLI, or equivalent external-CLI turn.
